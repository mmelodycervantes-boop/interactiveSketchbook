Reading Reflection 1

1. Affordances & Signifiers

In reviewing my micro-project “Terms and Conditions,” I realized that there are a few affordances, but mainly two. One of the affordances is the ability to scroll within the text box at a slow and medium speed. This affordance is signaled through a text underneath the accept button stating, “Use slow or medium scrolling to continue. Reach the end to unlock acceptance.” The other affordance is the ability to accept the terms and conditions. There are two main signals for this. There is also a warning that indicates that the user must “read the terms and conditions to accept,” and the accept button is greyed out, “locked,” therefore telling the user that they can not click the button, but rather have to do something. When the button turns green, it signals to the user that they are able to click the terms and conditions. I think that the affordances are clear due to the prompting, warning, and color change.

2. Feedback

In consideration of feedback, the system does a good job of giving the user feedback. When the system detects that the user is scrolling through the text at a fast speed, it immediately gives the user a warning to read the terms and conditions in order so that the accept button may become clickable. However, now looking back at the warning, I could have written something different to make it more understandable, such as “Warning! Skimming has been detected. Please read the terms and conditions to accept.” That way, the user knows exactly why they were given a warning. On the other hand, if the user continues to scroll fast, they are immediately given a 5-second timeout, which tells the user that scrolling has been paused. Instead of saying scrolling has been paused, I could have made the system say, “Uneligible to scroll. Please, read the terms and conditions.” I also think that once the user accepts the terms and conditions, the feedback given can be confusing. Instead of the button turning from Accept to Accepted, a pop-up near the URL shows up. Up until that point, the webpage would change, so I think Norman would have criticized that part of the design, since it goes against what the user would expect.

3. Mapping

I believe the system does show a clear relationship between input and outcome. For example, the box constraint for the text signals that the mouse can only move the text up or down within the box limits. Also, if the accept button says “locked” and is grey, it tells the user that they can’t click it yet. I think the relationship is when it comes to the scroll bar. A user might expect to be able to use the scroll bar to scroll up or down the text, but the bar does not give the user that ability. Instead, the scroll bar simply indicates where the user is within the text and if they have reached the bottom yet.

4. Constraints

The constraints that exist in my system are both visible and invisible. An invisible constraint is that the user only has the ability to scroll through the text within the text box; if they go anywhere outside the text box to scroll, the text will not move. A visible constraint is when the user is in a time-out from scrolling, which is communicated to the user through a countdown and information given through text. Another visible constraint is the locked button, which is communicated through the grey color and the word “locked.” I believe these constraints guide the user to understanding how to use the system; the system itself is designed to frustrate the user the more they try to avoid reading the text.

5. The Biggest Design Weakness

If Norman evaluated my project, I think he would say that my biggest weakness is that the confirmation for the accept button is not very clear. When the user accepts, the system says “melodycervantes-boop.github.io says Terms Accept. “Ok.” If I were a user, I would not be satisfied with the confirmation. I would have probably thought, “all that effort for this?” Which I think Norman would have also thought. It also just goes against the design of the system. It doesn’t seem as if it’s part of the system, but rather separate from it. Plus, it says my name, and if the user didn’t know me, it would have probably confused them.

6. What is Your Biggest Takeaway from the Text?

I actually enjoyed the reading, I thought it was very interesting as to how Norman sees the world in a design usability sense. I think what I related to the most was the problem with how the doors operate. I am one of those people who does read the signs of how they operate, but rather uses their intuition. I thought it was pretty funny how he talked about it. But I will admit, I never really thought about how perhaps it’s not our fault if we don’t know how something works or functions, but rather the designer themselves for not wondering why the users are using the design in a certain way. I think my biggest takeaway from this is that good design should anticipate human error and should have clear communication with the users.
